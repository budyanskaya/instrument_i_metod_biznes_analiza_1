# Инструменты и методы бизнес-анализа
## Задание 1
Используя нотацию BPMN и Bizagi Modeler, сформируйте и продемонстрируйте модель бизнес-процессов  TO-BE для автоматизации работы с клиентами в онлайн школе. Выполните анализ улучшений модели TO-BE (по сравнению с моделью AS-IS) по экономическим параметрам и по степени ее готовности для создания сценария интеграции.

## Решение
## AS IS 
<img width="1102" height="643" alt="image" src="https://github.com/user-attachments/assets/83e4844b-9398-4c2e-9ce0-85c3ae222a2f" />
Модель AS-IS описывает текущее состояние бизнес-процесса до внедрения комплексной автоматизации и CRM-системы. Процесс характеризуется высокой степенью вовлечения человеческого фактора, дублированием функций и отсутствием сквозного контроля данных.
Основные роли и зоны ответственности (Дорожки):
- Клиент: инициирует процесс через форму, самостоятельно оплачивает счета и ожидает доступы.
- Менеджер по продажам: выполняет функции оператора данных, осуществляет звонки, вручную формирует ссылки на оплату и контролирует процесс регистрации.
- Бухгалтер: производит ручную верификацию транзакций по банковским выпискам.
- Куратор / Преподаватель: осуществляет ручной онбординг студента в учебные чаты.

Сравнительный анализ выполнен по ключевым экономическим показателям эффективности (KPI): затраты времени (Time), финансовые издержки и ФОТ (Cost), а также качество процесса и минимизация рисков (Quality)

| Параметр анализа | Модель AS-IS (Ручная обработка) | Модель TO-BE (Автоматизированная) |
| :--- | :--- | :--- |
| **Скорость обработки (Time)** | От 2 до 24 часов. Зависит от графика работы сотрудников, скорости обработки выписок бухгалтером и ручного создания ссылок. | До 3–5 минут. Вебхуки с сайта мгновенно передают лид в CRM, платежная ссылка генерируется автоматически. |
| **Трудозатраты и издержки (Cost)** | Высокие (линейный рост). Увеличение потока студентов требует пропорционального расширения штата менеджеров для выполнения рутинных операций. | Минимальные (эффект масштаба). CRM берет на себя 80% рутины. Персонал привлекается только для качественного сервиса и сложных продаж. |
| **Ошибки и риски (Quality)** | Высокий человеческий фактор. Риск опечаток в email при создании аккаунта, потеря заявок в Excel, задержки отправки писем. | Близко к 0%. Передача данных между системами (Сайт → CRM → Эквайринг → GetCourse) стандартизирована по API. |
| **Доступность сервиса** | Ограничена рабочими часами персонала. Заявки, поступившие ночью, обрабатываются только утром, что снижает конверсию. | 24/7/365. Оплата, регистрация и отправка доступов происходят полностью автоматически в любое время суток. |

Автоматизация процесса по модели TO-BE превращает обработку лида из переменной затраты (зависящей от ФОТ) в фиксированную ИТ-инфраструктурную операцию. Это позволяет школе кратно масштабировать выручку без раздувания штата бэк-офиса.

## TO BE
Модель TO-BE обладает высоким уровнем интеграционной готовности (около 95%). Процесс полностью декомпозирован на атомарные шаги, каждый из которых имеет четкие входные и выходные цифровые данные (digital footprints).
Обоснование технической готовности:
- В модели полностью ликвидированы «информационные разрывы» (когда данные из одной системы нужно переносить в другую визуально-ручным способом).
- Четко разграничены автоматические системные действия (роботы, триггеры) и человеческий сервис (приветственный звонок куратора).
- Схема подготовлена к прямой трансляции в логику No-Code интеграторов (Make, Albato, Zapier) или написанию ТЗ для backend-разработчиков.

На BPMN-диаграмме «Онлайн-школа (TO-BE)» описан автоматизированный процесс работы с клиентами: клиент оставляет заявку, система регистрирует лида и проводит скоринг, затем либо автоматически генерирует ссылку на оплату, либо передаёт клиента менеджеру для консультации. После оплаты система создаёт аккаунт в LMS, куратор звонит клиенту, и клиент начинает обучение.

По сравнению с AS-IS модель TO-BE даёт экономию: сокращение времени обработки заявок на 80-90%, снижение операционных расходов на 60-70%, повышение конверсии в оплату на 15-25% за счёт автоматизации скоринга, генерации ссылок, создания аккаунта и чёткого цикла обработки возражений.

Готовность модели для интеграции — 4 из 5. Есть чёткие зоны ответственности, автоматические шаги и временной таймаут, но не хватает обработки ошибок интеграции (сбой LMS, недоступность платежей), протоколов обмена и длительности задач. Модель пригодна для ТЗ, но требует доработки.
